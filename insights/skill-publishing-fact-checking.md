## 概述

从零开发debunk（事实核查）技能并发布到ClawHub的完整实践，包括技能设计、内容验证、元数据配置和发布流程，以及实际应用案例。

## 背景

### 需求识别

**用户场景：**
- 老板/同事分享公众号/视频号内容要求辟谣
- 家庭群中的健康、科技等领域虚假信息
- 需要快速验证AI模型相关宣传的真实性

**技术挑战：**
- 公众号反爬机制破解
- 多源信息交叉验证
- 评分体系和可信度分级
- 回复话术的情商把控

### 技能架构设计

**文件结构：**
```
~/.openclaw/skills/debunk/
├── SKILL.md
├── README.md
├── SKILL-PUBLISHING.md
└── repository/ (可选)
```

## 技能开发实践

### SKILL.md核心设计

**输入处理：**
```markdown
## 场景定义
场景A：公众号链接核查
场景B：截图内容核查  
场景C：文字内容核查
```

**验证流程：**
1. **信息提取**：获取原文/链接内容
2. **多源验证**：搜索+独立评测+官方数据
3. **评分分级**：⭐⭐⭐⭐☆ 大部分属实，个别措辞有营销成分
4. **核查清单**：逐条验证核心论点
5. **锐评总结**：客观评价报道倾向
6. **回复话术**：提供多种风格选项

### 实际应用案例

**案例：阿里Qwen3.6-Plus性能核查**

**核查过程：**
1. **链接获取**：公众号文章反爬处理
2. **信息提取**：提取核心论点（"直逼Claude"、"国产最强"等）
3. **多源验证**：
   - renovateqr评测数据
   - 智东西实测报告
   - OpenRouter性能数据
4. **问题识别**：
   - "直逼Claude"部分属实（Terminal-Bench 61.6 > Claude 59.3）
   - "超越2-3倍参数量"无法验证（参数量未公开）
   - 实际存在26%代码推理幻觉率被忽略

**评分结果：**
⭐⭐⭐⭐☆ 大部分属实，个别措辞有营销成分

## ClawHub发布规范

### 元数据要求

**强制字段：**
```yaml
repository: https://github.com/username/skill-name
clawhub: https://clawhub.com/skill/name
```

**发布流程：**
1. **内容准备**：确保SKILL.md完整
2. **元数据检查**：验证repository和clawhub字段
3. **本地测试**：验证功能正常
4. **发布命令**：`clawhub publish`
5. **版本更新**：修复问题后更新版本号

### 展示名称规范

**格式要求：**
- 采用「Title Case（中文名称）」格式面向中文用户
- 示例：`Workflow Note（流程笔记）`

**问题修复：**
- 原名称：`workflow-note`
- 修正后：`Workflow Note（流程笔记）`
- 版本更新：`1.2.2`

### 安全信息处理

**敏感信息泄露事件：**
- **问题**：发布后发现SKILL.md中包含"不暴露Ty Chen真实身份"等私人信息
- **处理**：
  1. 立即删除包含私人信息的版本
  2. 发布修正版 `workflow-note@1.2.1`
  3. 同步更新SKILL-PUBLISHING.md规范

## 技能验证与优化

### 核查方法论

**可信度评级标准：**
- ⭐⭐⭐⭐⭐ 完全属实，数据充分
- ⭐⭐⭐⭐☆ 大部分属实，个别措辞有营销成分
- ⭐⭐⭐☆☆ 部分属实，存在明显误导
- ⭐⭐☆☆☆ 大部分虚假，少量事实
- ⭐☆☆☆☆ 基本虚假，纯属误导

**信息源优先级：**
1. 官方数据/发布渠道
2. 独立评测机构
3. 行业权威媒体
4. 社区讨论/用户反馈
5. 相关技术文档

### 回复话术设计

**风格选项：**
- 🔪 直言不讳版：犀利、实事求是
- 🤝 高级情商版：平衡客观与体面
- 🍬 阿谀奉承版：迎合讨好，不提问题
- 💤 敷衍了事版：通用短回复

**执行经验：**
- Skill结构明确要求提供多种风格
- 实际执行时漏掉可选部分
- 记录错误并反馈到学习系统

## 发布后维护

### 版本管理

**版本号规则：**
- 语义化版本：主版本号.次版本号.修订号
- 示例：`1.2.1`（修复问题）、`1.2.2`（名称规范）

**更新流程：**
1. 问题发现 → 本地修复
2. 测试验证 → 确认功能正常
3. 版本号递增 → 发布新版本
4. ClawHub同步 → 自动更新

### 用户反馈处理

**问题识别：**
- Skill结构理解偏差（漏掉可选部分）
- 元数据配置疏忽（repository/clawhub字段）
- 敏感信息泄露（私人信息暴露）

**改进措施：**
- 加强执行自检，确保Skill结构完整
- 建立发布前检查清单
- 明确安全边界，区分可分享内容和私人信息

## 经验总结

### 开发原则

1. **结构完整性**：Skill输出结构必须严格遵循"缺一不可"
2. **安全性**：明确区分可分享内容和私人信息
3. **用户导向**：提供多种风格选项，适应不同场景

### 发布规范

1. **元数据必须**：repository和clawhub字段是强制要求
2. **版本管理**：语义化版本号，记录变更内容
3. **展示规范**：中英文混合的Title Case格式

### 实用技巧

1. **多源验证**：单一信息源不可靠，需要交叉验证
2. **评分透明**：明确评分标准，避免误导
3. **风格选择**：根据用户关系选择合适的回复风格
