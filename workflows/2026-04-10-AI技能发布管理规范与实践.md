# AI技能发布管理规范与实践

## 背景

随着AI技能数量的增加，建立一个标准化的发布管理流程变得尤为重要。本篇记录了从2026年3月底到4月初的技能发布实践经验，包括发布流程、元数据规范、质量控制等关键环节。

## 技能发布流程

### 1. 开发阶段规范

#### 项目结构标准
```markdown
~/.openclaw/skills/[skill-name]/
├── SKILL.md              # 技能文档（必须）
├── README.md             # 项目说明（可选）
├── scripts/             # 工具脚本
├── references/          # 参考资料
├── examples/            # 使用示例
└── tests/               # 测试用例
```

#### 文档编写规范
- **必须包含**：功能描述、使用方法、参数说明、注意事项
- **推荐包含**：使用场景、示例代码、常见问题、更新日志
- **禁止内容**：个人信息、敏感信息、过时信息

### 2. 元数据规范

#### 强制元字段
根据ClawHub发布要求，每个技能必须包含以下元数据：

```yaml
---
# SKILL.md frontmatter
title: 技能名称
description: 技能功能描述
version: "1.0.0"
repository: https://github.com/用户名/仓库名称
clawhub: https://clawhub.com/用户名/技能名称
author: Tino Chen
category: [tools, workflows, insights]
tags: [tag1, tag2, tag3]
---
```

#### 元数据验证清单
发布前必须检查：
- [ ] repository字段存在且有效
- [ ] clawhub字段存在且指向ClawHub页面
- [ ] 版本号符合语义化版本规范
- [ ] 分类标签准确
- [ ] 无敏感信息泄露

### 3. 发布流程

#### GitHub发布
```bash
# 1. 确保代码完整性
git add .
git commit -m "feat: 发布技能版本x.x.x"
git tag vx.x.x
git push origin main
git push origin vx.x.x

# 2. 创建GitHub Release
gh release create vx.x.x --title "技能名称 vx.x.x" --notes "更新内容"
```

#### ClawHub发布
```bash
# 1. 使用clawhub CLI发布
clawhub publish

# 2. 验证发布结果
clawhub技能名称
```

#### 展示名称规范
面向中文用户的展示名称必须采用「Title Case（中文名称）」格式：
- 正确：Workflow Note（流程笔记）
- 错误：workflow-note
- 错误：WorkflowNote

### 4. 版本管理规范

#### 语义化版本
```markdown
主版本号：不兼容的API修改
次版本号：向下兼容的功能性新增
修订号：向下兼容的问题修正

示例：
1.0.0 -> 1.0.1：bug修复
1.0.0 -> 1.1.0：新增功能  
1.0.0 -> 2.0.0：不兼容更新
```

#### 版本日志规范
```markdown
## [1.1.0] - 2026-04-06

### 新增
- 支持XX功能
- 增加YY配置选项

### 优化
- 改进ZZ性能
- 优化文档说明

### 修复
- 修复了ABC问题
- 解决了DEF异常
```

## 实践案例

### 案例1：敏感信息泄露修复

#### 问题发现
2026年4月6日，在发布`workflow-note`技能时发现SKILL.md中包含敏感信息：
```markdown
# 错误示例
"不暴露用户真实身份"
```

#### 解决方案
1. 立即撤回发布
2. 删除敏感信息
3. 替换为通用规则：
```markdown
# 正确示例
"保护用户隐私，不泄露个人身份信息"
```
4. 重新发布版本1.2.1

#### 经验教训
- 发布前必须仔细检查所有文档内容
- 避免包含任何个人信息或敏感配置
- 建立敏感信息检查清单

### 案例2：展示名称规范修复

#### 问题发现
审核发现ClawHub展示名称不符合规范，缺少中文名称。

#### 解决方案
1. 更新SKILL.md frontmatter：
```yaml
title: "Workflow Note（流程笔记）"
```
2. 发布版本1.2.2
3. 更新SKILL-PUBLISHING.md中的对照表

#### 经验教训
- 展示名称面向中文用户需要完整的中英双语
- 保持与现有规范的统一性
- 及时更新相关文档

### 案例3：元数据字段调查

#### 诡异发现
在重构`daily-ai-briefing`技能时，发现`isolated: true`字段在技能系统中**无实际作用**。

#### 调查过程
```markdown
1. 搜索OpenClaw源码（dist/目录）
   - 未找到读取isolated字段的代码
2. 搜索ClawHub CLI源码
   - 未找到相关处理逻辑
3. 分析技能加载流程
   - 结论：该字段被忽略，仅存在于cron的sessionTarget中
```

#### 经验教训
- 不要依赖文档中未明确的配置字段
- 技能系统实际行为可能与文档不符
- 需要亲自验证配置的实际效果

## 发布策略

### 1. 版本发布节奏

#### 重大版本（x.0.0）
- 包含架构变更
- 可能存在不兼容更新
- 需要充分测试和文档更新

#### 功能版本（x.x.0）
- 新增重要功能
- 向下兼容
- 需要功能测试和文档更新

#### 修复版本（x.x.x）
- bug修复和优化
- 向下兼容
- 可以快速发布

### 2. 发布时间选择

```markdown
✅ 推荐发布时间：
- 工作日上午10:00-11:00
- 工作日下午15:00-16:00
- 周末上午9:00-10:00

❌ 避免发布时间：
- 深夜23:00-次日8:00
- 工作高峰期（9:00-10:00, 14:00-15:00）
- 重要节假日前后
```

### 3. 回滚策略

#### 问题识别
```markdown
🔴 严重问题（必须回滚）：
- 功能完全不可用
- 安全漏洞
- 数据丢失风险
- 大量用户投诉

🟡 中等问题（可修复后重新发布）：
- 功能异常
- 性能下降
- 文档错误
- 少数用户受影响
```

#### 回滚流程
```bash
# 1. 确认问题
git checkout v1.0.0  # 回滚到上一个稳定版本
git tag -d v1.0.1   # 删除问题版本tag
git push origin --delete v1.0.1

# 2. 重新发布
git commit -m "revert: 回滚到v1.0.0"
git tag v1.0.1
git push origin main v1.0.1

# 3. ClawHub重新发布
clawhub publish
```

## 工具和脚本

### 1. 发布自动化脚本

```bash
#!/bin/bash
# scripts/publish-skill.sh

# 检查元数据
check_metadata() {
    if ! grep -q "repository:" SKILL.md || ! grep -q "clawhub:" SKILL.md; then
        echo "错误：缺少必要的元数据字段"
        exit 1
    fi
}

# 格式检查
format_check() {
    if ! command -v markdownlint &> /dev/null; then
        echo "警告：未安装markdownlint，跳过格式检查"
    else
        markdownlint *.md
    fi
}

# 发布流程
publish_skill() {
    check_metadata
    format_check
    
    # 提交代码
    git add .
    git commit -m "feat: 发布版本$1"
    git tag "v$1"
    git push origin main "v$1"
    
    # GitHub Release
    gh release create "v$1" --title "$(grep '^title:' SKILL.md | cut -d' ' -f2-) v$1" --notes "版本更新内容"
    
    # ClawHub发布
    clawhub publish
}

# 执行发布
publish_skill "$1"
```

### 2. 元数据验证工具

```python
#!/usr/bin/env python3
# scripts/validate_metadata.py

import yaml
import re
import sys
from pathlib import Path

def validate_skill_metadata(skill_path):
    """验证技能元数据"""
    skill_md = Path(skill_path) / "SKILL.md"
    
    with open(skill_md) as f:
        content = f.read()
    
    # 提取frontmatter
    frontmatter_match = re.search(r'^---\s*\n(.*?)\n---\s*\n', content, re.MULTILINE | re.DOTALL)
    
    if not frontmatter_match:
        print("❌ 缺少frontmatter")
        return False
    
    try:
        metadata = yaml.safe_load(frontmatter_match.group(1))
    except yaml.YAMLError as e:
        print(f"❌ frontmatter格式错误: {e}")
        return False
    
    # 检查必需字段
    required_fields = ['title', 'description', 'version', 'repository', 'clawhub']
    
    for field in required_fields:
        if field not in metadata:
            print(f"❌ 缺少必需字段: {field}")
            return False
    
    # 检查repository格式
    if not metadata['repository'].startswith('https://github.com/'):
        print("❌ repository格式错误")
        return False
    
    # 检查clawhub格式
    if not metadata['clawhub'].startswith('https://clawhub.com/'):
        print("❌ clawhub格式错误")
        return False
    
    print("✅ 元数据验证通过")
    return True

if __name__ == "__main__":
    skill_path = sys.argv[1] if len(sys.argv) > 1 else "."
    validate_skill_metadata(skill_path)
```

## 经验教训

### 成功经验
1. **元数据标准化**：统一的标准显著提高了发布效率
2. **自动化工具**：脚本验证减少了人为错误
3. **质量分级**：不同级别的问题采用不同的处理策略
4. **文档同步**：保持文档与代码的一致性

### 失败教训
1. **信息泄露**：差点在文档中暴露敏感信息
2. **名称规范**：展示名称不符合用户期望
3. **字段误解**：依赖了实际不存在的配置字段

### 未来改进

2. **测试覆盖**：提高自动化测试覆盖率

## 总结

技能发布管理是一个系统工程，需要从开发、测试、发布到维护的全流程管理。通过建立标准化的流程、使用自动化工具、严格的质量控制，可以显著提高发布效率和产品质量。

最重要的原则是：**质量第一、用户至上、持续改进**。每一个技能的发布都应该追求卓越的质量和用户体验，同时通过持续学习和改进来不断提升发布管理水平。
