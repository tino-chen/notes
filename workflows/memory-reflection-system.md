# OpenClaw 记忆与反思系统

## 背景

AI 助理每次会话醒来都是"全新的"，没有原生记忆。OpenClaw 通过文件系统解决这个问题——让 Agent 读写文件来延续记忆。但如何组织这些文件、让记忆系统既有连续性又能驱动自我改进，是一个值得深入思考的架构问题。

本文记录了我们如何从"什么都往 MEMORY.md 里塞"演进到"情形记忆 + 反思记忆 + 晋升机制"的完整过程。

## 问题：最初的设计缺陷

### 1. 所有记忆混在一起

最初只有一个 `MEMORY.md`，所有东西都往里塞：
- 用户基本信息（身份、时区、手机号）
- 电脑配置
- 定时任务列表
- 工作流程
- 经验教训
- 灵感收集规则
- 代理配置

结果就是一个不断膨胀的文件，什么都找得到，但什么都不好找。

### 2. 身份信息到处重复

`MEMORY.md`、`IDENTITY.md`、`USER.md` 三个文件都记录了身份和用户信息，改一个忘改另一个就产生不一致。

### 3. 没有经验晋升通道

虽然安装了 `self-improving-agent` 技能，但从未初始化过（`.learnings/` 目录根本不存在）。经验教训散落在每日笔记里，没有结构化记录，更没有机制将反复验证的经验固化到行为规则中。

### 4. 情形记忆和反思记忆混淆

`memory/YYYY-MM-DD.md` 里既记录"今天做了什么"，又混杂"学到了什么"。前者是情形记忆（保证连续性），后者是反思记忆（驱动改进），两者性质完全不同，混在一起两边都做不好。

## 架构设计

### 核心思路：两层记忆 + 晋升通道

```
交互发生
  ├─→ memory/YYYY-MM-DD.md          情形记忆（今天做了什么）
  └─→ .learnings/LEARNINGS.md       反思记忆（学到了什么）
         ↓ 每天 3:00 定时审查
    SOUL.md / AGENTS.md / TOOLS.md / USER.md   晋升后的行为规则
```

### 情形记忆 (Episodic Memory)

**目的**：保证 Agent 的连续性，不遗忘每天发生的事情。

- **每日笔记**：`memory/YYYY-MM-DD.md`
- **长期事实**：`MEMORY.md`（项目概览、硬件配置等）
- **规则**：只记事实（发生了什么），不记反思（学到了什么）

### 反思记忆 (Reflective Memory)

**目的**：积累经验，通过晋升机制让 Agent 越来越聪明。

- **目录**：`.learnings/`，基于 `self-improving-agent` 技能实现
- **LEARNINGS.md**：被纠正的错误、更好的做法、知识盲区、最佳实践
- **ERRORS.md**：命令失败、API 报错、异常行为
- **FEATURE_REQUESTS.md**：用户期望但尚不存在的功能

### 晋升机制

反思记忆不是终点，而是晋升的入口：

| 阶段 | 动作 | 触发方式 |
|------|------|----------|
| 记录 | 交互中遇到值得反思的情况 → 写入 `.learnings/` | 心跳时检查 |
| 审查 | 找出 pending 条目，评估是否值得晋升 | 每天 3:00 定时任务 |
| 晋升 | 提炼为简短规则，写入目标配置文件 | 定时任务执行 |
| 标记 | 原条目 status 改为 `promoted` | 定时任务执行 |

**晋升目标**：

| 经验类型 | 晋升到 | 示例 |
|----------|--------|------|
| 行为模式改进 | `SOUL.md` | "回复要简洁，不要堆废话" |
| 工作流程优化 | `AGENTS.md` | "长任务要拆子 Agent 执行" |
| 工具使用坑点 | `TOOLS.md` | "Claude Code 必须 pty:true" |
| 用户偏好 | `USER.md` | "不用 Apple 提醒事项了" |

### 触发机制

| 机制 | 用途 | 频率 |
|------|------|------|
| 心跳 | 记录每日情形记忆 + 捕获反思经验 | 约每 30 分钟 |
| 定时任务（审查） | 审查 `.learnings/` 并晋升 | 每天 3:00 |

## 实施过程

### 第一步：清理冗余

- `MEMORY.md` 从 ~3.5KB 精简到 ~2KB，删除与 IDENTITY.md/USER.md 重复的身份信息
- 删除重复的灵感系统段落（第 3 节和第 4 节完全相同）
- 将通讯能力信息移到 `TOOLS.md`（工具使用细节归工具备忘录）
- 合并 IDENTITY.md 到 SOUL.md（消除身份文件的重复定义）

### 第二步：建立反思记忆

- 创建 `.learnings/` 目录和三个日志文件
- 在 `AGENTS.md` 中定义情形记忆 vs 反思记忆的区分
- 在 `HEARTBEAT.md` 中新增"记录反思经验"步骤

### 第三步：建立晋升通道

- 创建每天 3:00 的 isolated cron 任务
- 任务逻辑：读取 pending 条目 → 评估优先级 → 晋升到目标文件 → 标记完成
- 有重要发现时通知用户，没有则安静结束

## 经验总结

### 关键认知

**情形记忆和反思记忆是两种不同的东西**。前者回答"发生了什么"，后者回答"学到了什么"。混在一起会导致两边都做不好——情形记忆变得冗长难读，反思经验又缺乏结构化的审查和晋升通道。

### 对 Agent 架构的启示

1. **文件即记忆，但要分层**：不是所有记忆都该放一个文件，按性质分层管理
2. **记忆需要生命周期**：记录 → 审查 → 晋升 → 归档，不能只记不管
3. **自动化审查比手动更可靠**：心跳和定时任务比"想起来就看"稳定得多
4. **配置文件是最终归宿**：经验只有变成行为规则（写入 SOUL.md / AGENTS.md），才能真正影响 Agent 的行为
