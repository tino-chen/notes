# 每日AI简报系统架构重构实践

## 概述

从单技能单体架构重构为双技能分离架构的完整实践，解决了GLM-5-turbo模型在复杂多阶段任务中的"偷懒"问题，建立了可靠的质量控制机制。

## 背景

原系统采用单一`daily-ai-briefing`技能，存在三个核心问题：

1. **模型行为问题**：GLM-5-turbo在单会话内处理三阶段任务时，读完文件汇报计划就终止，不执行工具调用
2. **技术限制**：`sessions_spawn`在cron会话中不适用，子会话结果无法返回主会话
3. **复杂度过高**：SKILL.md超300行，模型迷失在元信息中

### 触发事件

2026年4月6日，质量审查中发现简报存在严重问题：
- **跨天重复**：前日已报道的内容在当日重复出现
- **来源失效**：文章重定向到地区不可用页面
- **信息要素缺失**：缺少When/Where/Who等关键要素
- **审核失效**：全部内容不合规，审核环节形同虚设

## 问题诊断

### 根因分析

1. **审核机制缺陷**
   - 仅做文字审阅，不做独立验证
   - 未读取历史简报进行去重
   - 未web_fetch来源链接验证
   - 未搜索确认事件时间真实性

2. **搜索策略问题**
   - 查询词过于宽泛，导致结果重复
   - 缺乏领域细分和去重机制
   - 依赖单一新闻源，缺乏多样性

3. **系统架构局限**
   - 三阶段设计在实际运行中过早终止
   - 会话管理问题导致Stage 3修正未执行
   - 技能文件过长导致模型迷失

### 发现的非显而易见问题

**1. GLM-5-turbo的agentic loop过早终止**
- 现象：不是maxTokens限制（配置8192），是模型自己选择停止生成
- 影响：三阶段任务的第三阶段（修正）无法执行
- 解决：拆分为独立技能，每阶段独立会话

**2. 代理连接稳定性问题**
- 现象：IMAP连接频繁`connection reset by peer`
- 根因：Clash代理无法处理IMAP（端口993）协议
- 解决：改用gog（OAuth over HTTPS API）

**3. 技能元数据字段误解**
- 调查：frontmatter中`isolated: true`字段在OpenClaw源码中无实际作用
- 结论：该概念只存在于cron任务的`sessionTarget`属性中

## 重构方案

### 1. 质量控制机制重构

#### 红线分级制度

将质量红线从"一刀切删除"改为分级处理：

**🔴 致命问题（必须删除）：**
- 14天内重复内容
- 明确虚假信息
- 来源失效链接
- 超30天事件（除非重大突破）

**🟡 可修正问题（打回修正）：**
- 要素缺失（When/Where/Who）
- 模糊表述（需要具体化）
- 分类错误（需要重新归类）
- 审核过严导致空简报
- 域名霸榜（单域名超3条）

#### 自检流程优化

**搜索策略升级：**
- 按领域细分查询词
- 域名去重（每域名最多3条）
- Review评分机制（1-10分）
- 兜底保留（全部毙掉时按分数保留1-3条）
- 空简报推送通知

**自检输出改进：**
- 强制输出JSON格式自检报告
- 从"默答"改为结构化输出
- 删除硬编码日期（避免过时问题）

### 2. 搜索策略优化

#### sources配置重构

将sources.md从"流程+配置混合"精简为纯配置：

| 领域 | 搜索引擎 | 查询策略 | 兜底策略 |
|------|---------|---------|---------|
| 学术论文 | Semantic Scholar | 关键词+时间范围(7-30天) | arXiv热门 |
| GitHub | GitHub API | stars:>100+push:>10+created:>7d | GitHub Trending |
| 行业动态 | 8个新闻源 | 领域细分+域名去重 | 科技媒体头条 |
| 媒体锐评 | Brave Search | "专家观点"+时间范围 | 行业领袖分析 |

**搜索源扩展**：从3个扩展到8个（techcrunch/venturebeat/theverge/arstechnica/36kr/jiqizhixin/qbitai/infoq）

#### 搜索深度策略
- **第一层**：基础搜索（sources.md配置）
- **第二层**：深度搜索（检测结果驱动）
- **第三层**：兜底搜索（学术+GitHub+行业）

### 3. 系统架构拆分

#### 原架构问题

```
❌ 单技能三阶段设计：
- Stage 1: 生成
- Stage 2: 审核
- Stage 3: 修正推送

问题：
- GLM-5-turbo在单会话内反复"偷懒"
- sessions_spawn在cron会话中不适用
- SKILL.md过长(300+行)导致模型迷失
```

#### 新架构方案

拆分为两个独立技能，**只在流程上区别**：

- `daily-ai-briefing`（轻量版，~150行）：单阶段生成+自检→推送
- `agentic-daily-ai-briefing`（重量版，~180行）：三阶段（生成→审核→修正推送）

**关键设计原则：**
- 轻重版本只在流程上差异，内容标准完全一致
- 共享`sources.md`和数据目录，避免重复配置
- 删除无用的`isolated: true` frontmatter（经源码确认该字段无实际作用）

#### 代码组织

```
daily-ai-briefing/
├── SKILL.md (~150行，轻量版)
├── agentic-daily-ai-briefing/
│   └── SKILL.md (~180行，重量版)
├── sources.md (共享配置)
└── data/ (共享数据目录)
```

#### Cron任务配置

```yaml
# 轻量版（当前启用）
- schedule: "0 8 * * *"
  sessionTarget: "isolated"
  payload: { kind: "agentTurn", message: "生成今日简报" }

# 重量版（当前禁用）
- schedule: "0 8 * * *"
  sessionTarget: "isolated"
  payload: { kind: "agentTurn", message: "生成今日简报" }
- schedule: "0 8 10 * * *"
  sessionTarget: "isolated"
  payload: { kind: "agentTurn", message: "审核今日简报" }
- schedule: "0 8 20 * * *"
  sessionTarget: "isolated"
  payload: { kind: "agentTurn", message: "修正推送今日简报" }
```

## 实施过程

### 第一阶段：质量机制优化
1. **红线分级**：建立🔴/🟡两级质量标准
2. **去重机制**：实现语义级别的实体事件比对
3. **自检输出**：强制输出JSON格式的自检报告
4. **板块定义**：明确边界案例，避免分类歧义

### 第二阶段：内容格式规范
1. **事件/锐评分类**：
   - 事件报道：`据xxx媒体x月x日报道`
   - 媒体锐评：`锐评：xxx媒体认为`
2. **删除硬编码日期**：避免"今天是4月4日"这类过时表述

### 第三阶段：系统拆分
1. 创建轻量版技能（daily-ai-briefing）
2. 创建重量版技能（agentic-daily-ai-briefing）
3. 配置相应的cron任务
4. 测试两个版本的运行效果

### 第四阶段：测试验证

```
测试结果：
✅ Stage 1+2 独立验证质量高
  - 成功拦截MiniMax Speech 2.8 HD虚假信息
  - 发现跨天重复内容
  - 拦截超14天条目
❌ Stage 3 修正未执行
  - 会话在Stage 2后提前结束
  - timeout或token耗尽
```

## 关键决策

### 为什么选择双技能方案
1. **模型行为特性**：GLM-5-turbo在复杂任务中过早终止
2. **会话限制**：cron环境下的sessions_spawn不适用
3. **维护成本**：长技能文件难以维护和调试
4. **灵活性**：允许用户选择质量vs效率的平衡

### 为什么保留内容标准
- 轻量版和重量版的**内容要求完全一致**
- 区别只在**执行流程**
- 确保输出质量不受流程简化影响

## 实施效果

### 质量提升
- 生成结果被新闻源霸榜问题 → 域名去重解决
- 审核过严导致空简报 → 评分机制+兜底保留
- 跨天重复和超期内容 → 红线拦截机制
- 成功拦截了MiniMax M2.5的虚假信息（违反14天红线）

### 系统稳定性
- 代理连接不稳定 → 切换到OAuth API
- 模型过早终止 → 架构拆分解决

### 运维效率
- 轻量版和重量版独立部署
- 共享配置减少维护成本
- 清晰的错误分级便于快速响应

## 经验总结

### 设计原则
1. **轻量是流程的轻量，不是内容要求的轻量**
2. **技能要保持自身独立，不要相互引用**
3. **配置文件只写配置，流程逻辑放SKILL.md**

### 质量控制
1. **审核机制应区分可优化问题和必须删除的严重问题**
2. **遇到问题第一反应不是重试，而是反思为什么出现，优化后再重试**
3. **自检步骤必须严格执行，不能假设存在**

### 系统设计
1. **架构简单性**：有时候拆分比整合更有效
2. **配置管理**：清晰的配置文件是系统可维护性的基础
3. **文档一致性**：代码实际行为可能与文档不符，需要验证

### 工具选择
1. **优先HTTPS API**（如gog），避免不稳定连接（如himalaya的proxychains4）
2. **安装即验证**：工具/技能安装后必须检查初始化状态
3. **代理优先HTTPS**：外部工具访问优先使用HTTPS代理

## 后续优化方向

1. **性能优化**：进一步减少重复搜索，提高效率
2. **扩展性**：支持更多新闻源和搜索策略
3. **用户体验**：优化推送格式和提醒机制
4. **监控告警**：增加质量监控和异常告警
