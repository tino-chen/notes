# 每日AI简报系统架构重构实战

## 背景

每日AI硬核进展简报系统自上线以来，经历了多次架构调整和优化。本篇记录了从2026年4月初开始的重大重构过程，展示了如何面对质量问题进行系统性思考和架构升级。

## 问题诊断

### 初始质量问题

2026年4月6日，质量审查中发现简报存在严重问题：
- **Gemma 4 跨天重复**：4/5已报道的内容在4/6重复出现
- **来源失效**："Claude 上下文窗口政策"文章重定向到地区不可用页面
- **信息要素缺失**："Claude Code 功能更新"缺少When要素
- **审核失效**：三条简报内容全部不合规，审核环节形同虚设

### 根因分析

通过深入分析，发现问题的根本原因：

1. **审核机制缺陷**
   - 仅做文字审阅，不做独立验证
   - 未读取历史简报进行去重
   - 未web_fetch来源链接验证
   - 未搜索确认事件时间真实性

2. **搜索策略问题**
   - 查询词过于宽泛，导致结果重复
   - 缺乏领域细分和去重机制
   - 依赖单一新闻源，缺乏多样性

3. **系统架构局限**
   - 三阶段设计在实际运行中过早终止
   - 会话管理问题导致Stage 3修正未执行
   - 技能文件过长导致模型迷失

## 重构方案

### 1. 质量控制机制重构

#### 红线分级制度
将质量红线从"一刀切删除"改为分级处理：

```markdown
🔴 致命红线（必须删除）：
- 14天内重复内容
- 明确虚假信息
- 来源失效链接
- 超30天事件（除非重大突破）

🟡 可修正问题（打回修正）：
- 要素缺失（When/Where/Who）
- 模糊表述（需要具体化）
- 分类错误（需要重新归类）
```

#### 去重机制升级
从简单的名称匹配改为**实体+事件类型语义比对**：
```python
# 伪代码示例
def is_duplicate(new_event, existing_events):
    # 相同实体类型 + 相同事件类型 + 时间相近 = 重复
    if new_event.entity == existing.entity and \
       new_event.event_type == existing.event_type and \
       abs(new_event.timestamp - existing.timestamp) < timedelta(days=1):
        return True
    return False
```

### 2. 搜索策略优化

#### sources配置重构
将sources.md从"流程+配置混合"精简为纯配置：

```markdown
| 领域       | 搜索引擎       | 查询策略                     | 兜底策略     |
|------------|---------------|------------------------------|--------------|
| 学术论文   | Semantic Scholar | 关键词+时间范围(7-30天)       | arXiv热门    |
| GitHub     | GitHub API     | stars:>100+push:>10+created:>7d | GitHub Trending |
| 行业动态   | 8个新闻源     | 领域细分+域名去重           | 科技媒体头条 |
| 媒体锐评   | Brave Search   | "专家观点"+时间范围         | 行业领袖分析 |
```

#### 搜索深度策略
- **第一层**：基础搜索（sources.md配置）
- **第二层**：深度搜索（检测结果驱动）
- **第三层**：兜底搜索（学术+GitHub+行业）

### 3. 系统架构拆分

#### 原架构问题
```markdown
❌ 单技能三阶段设计：
- Stage 1: 生成
- Stage 2: 审核  
- Stage 3: 修正推送

问题：
- GLM-5-turbo在单会话内反复"偷懒"
- sessions_spawn在cron会话中不适用
- SKILL.md过长(300+行)导致模型迷失
```

#### 新架构方案
拆分为两个独立技能，**只在流程上区别**：

**daily-ai-briefing**（轻量版）：
- 单阶段：生成→自检→推送
- 文件长度：~150行
- 适用于：质量要求高、容错率低的场景

**agentic-daily-ai-briefing**（重量版）：
- 三阶段：生成→审核→修正推送
- 文件长度：~180行
- 适用于：需要多重质量保证的场景

#### 共享组件设计
两个技能共享的核心组件：
- `sources.md`（搜索源配置）
- `data/` 目录（历史数据和缓存）
- `config/` 目录（配置文件）
- `templates/` 目录（输出模板）

### 4. 技能元数据调查

#### 诡异发现
在重构过程中发现技能frontmatter中`isolated: true`字段在OpenClaw系统中**无实际作用**。

#### 调查过程
1. 搜索OpenClaw源码（dist/目录）
2. 搜索ClawHub CLI源码
3. 分析技能加载逻辑
4. 结论：该字段被忽略，只存在于cron任务的`sessionTarget`属性中

#### 经验教训
- 不要依赖文档中未明确的配置
- 技能系统实际行为可能与文档不符
- 需要亲自验证配置效果

## 实施过程

### 第一阶段：质量机制优化
1. **红线分级**：建立🔴/🟡两级质量标准
2. **去重机制**：实现语义级别的实体事件比对
3. **自检输出**：强制输出JSON格式的自检报告
4. **板块定义**：明确边界案例，避免分类歧义

### 第二阶段：内容格式规范
1. **事件/锐评分类**：
   - 事件报道：`据xxx媒体x月x日报道`
   - 媒体锐评：`锐评：xxx媒体认为`
2. **删除硬编码日期**：避免"今天是4月4日"这类过时表述

### 第三阶段：系统拆分
1. 创建轻量版技能（daily-ai-briefing）
2. 创建重量版技能（agentic-daily-ai-briefing）
3. 配置相应的cron任务
4. 测试两个版本的运行效果

### 第四阶段：测试验证
```markdown
测试结果：
✅ Stage 1+2 独立验证质量高
  - 成功拦截MiniMax Speech 2.8 HD虚假信息
  - 发现跨天重复内容
  - 拦截超14天条目
❌ Stage 3 修正未执行
  - 会话在Stage 2后提前结束
  - timeout或token耗尽
```

## 关键决策

### 为什么选择双技能方案
1. **模型行为特性**：GLM-5-turbo在复杂任务中过早终止
2. **会话限制**：cron环境下的sessions_spawn不适用
3. **维护成本**：长技能文件难以维护和调试
4. **灵活性**：允许用户选择质量vs效率的平衡

### 为什么保留内容标准
- 轻量版和重量版的**内容要求完全一致**
- 区别只在**执行流程**
- 确保输出质量不受流程简化影响

### Cron配置策略
```yaml
# 轻量版
- schedule: "0 8 * * *"
  sessionTarget: "isolated"
  payload: { kind: "agentTurn", message: "生成今日简报" }

# 重量版（禁用）
- schedule: "0 8 * * *"
  sessionTarget: "isolated" 
  payload: { kind: "agentTurn", message: "生成今日简报" }
- schedule: "0 8 10 * * *"
  sessionTarget: "isolated"
  payload: { kind: "agentTurn", message: "审核今日简报" }
- schedule: "0 8 20 * * *"
  sessionTarget: "isolated"
  payload: { kind: "agentTurn", message: "修正推送今日简报" }
```

## 经验教训

### 技术层面
1. **模型局限性**：需要理解不同模型在复杂任务中的行为特点
2. **会话管理**：cron环境下的会话管理有特殊限制
3. **组件复用**：共享组件设计能显著提高维护效率

### 流程层面  
1. **质量分级**：质量问题需要分级处理，不能一刀切
2. **测试验证**：每个重大变更都需要完整的测试验证
3. **用户反馈**：及时响应用户反馈，快速迭代优化

### 系统设计
1. **架构简单性**：有时候拆分比整合更有效
2. **配置管理**：清晰的配置文件是系统可维护性的基础
3. **文档一致性**：代码实际行为可能与文档不符，需要验证

## 结果验证

重构后的系统在后续测试中表现良好：
- 成功拦截了MiniMax M2.5的虚假信息（2月12日发布违反14天红线）
- 域名去重机制有效避免了新闻源霸榜问题
- 搜索查询按领域细分提高了结果相关性
- 自检机制显著提升了输出质量

## 后续优化方向

1. **性能优化**：进一步减少重复搜索，提高效率
2. **扩展性**：支持更多新闻源和搜索策略
3. **用户体验**：优化推送格式和提醒机制
4. **监控告警**：增加质量监控和异常告警

## 总结

本次架构重构是一个典型案例，展示了如何面对质量问题进行系统性思考和架构升级。通过深入分析根因、设计合理的拆分方案、保持内容标准一致性，最终实现了质量和效率的平衡。

最重要的是，我们建立了可持续的质量控制机制和架构设计原则，为后续的系统演进奠定了基础。
