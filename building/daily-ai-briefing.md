# 每日 AI 进展简报系统

## 背景与目标

希望每天早上醒来就能看到一份高质量的 AI 行业简报，覆盖重大研究、开源热点和行业动态。要求：

- **全自动**：不需要人工干预，定时生成、审查、推送
- **高质量**：信息准确、来源可靠、无营销话术
- **去重**：同一事件不会在不同天的简报中重复出现

## 技术方案

基于 OpenClaw 的 Cron + Isolated Session 架构，拆分为两个独立任务：

```
08:00  生成任务（isolated session）
       └─ 搜索 → 过滤 → 去重 → 生成简报 → 保存文件

08:10  审查任务（isolated session）
       └─ 读取简报 → 逐条审查红线 → 修正 → 推送飞书
```

### 为什么拆成两步？

1. **单一职责**：生成和审查是不同目标，混在一起容易顾此失彼
2. **质量保障**：审查任务专注于检查质量，不受生成过程的上下文干扰
3. **容错**：生成失败不影响审查，审查发现问题可以修正后推送

### 关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 模型 | GLM-5-Turbo | Agent 执行纪律强，工具调用链稳定；GLM-5.1 编程更强但多步任务不稳定 |
| Session | isolated | 不污染主会话上下文，独立执行 |
| Timeout | 600s | 多轮搜索 + 多次 web_fetch 耗时较长 |
| 存储 | 本地 md 文件 | 便于去重审查，也方便回溯历史 |
| 推送 | 飞书（OpenClaw announce） | 比直接发消息更可靠 |

## 踩坑记录

### 1. GLM-5.1 执行多步任务不稳定

最初用 GLM-5.1 做生成任务，频繁出现：
- 格式不遵循指令
- 工具调用链断裂（搜到一半就停了）
- 不执行文件写入（只输出不存盘）

**解决**：换用 GLM-5-Turbo，专为工具调用和长链路执行优化，Agent 场景更稳定。

### 2. 推送全部失败

前 4 天简报全部 `not-delivered`，原因是 cron job 的 `delivery.mode` 设成了 `"none"`。

**解决**：改为 `"announce"`，由 OpenClaw 自动将审查任务的输出推送到飞书。

### 3. 红线规则不清晰导致质量失控

初期简报出现各种问题：
- 收录了 1 月份的新闻（超过 14 天时效）
- Star 数写"⭐ 新增 Star 数"占位符（未实际获取）
- 收录了 awesome-list（规则禁止）
- 时间表述模糊（"近日""近期"）

**解决**：重写 SKILL.md，制定 7 条红线规则（一票否决），每条都有明确的判断标准和处理方式。

### 4. 跨天重复

同一事件（如 Claude Sonnet 4.6 发布）在 4/2 和 4/3 连续出现。

**解决**：生成前强制读取最近 7 天历史简报，列出所有已出现的项目名/事件名，逐条去重。

## 红线规则（核心质量保障）

简报系统最重要的设计是 **7 条红线规则**，任何一条违规就直接丢弃该条目：

1. **14 天时效**：事件超过 14 天 → 丢弃
2. **来源白名单**：来源不在白名单内 → 丢弃
3. **GitHub 真实性**：awesome-list/论文合集 → 丢弃
4. **Star 数真实**：占位符/估算值 → 丢弃
5. **四要素完整**：Who/When/What/Impact 缺一不可，When 禁止模糊表述
6. **日期可追溯**：来源本身模糊时，必须额外搜索确认具体日期
7. **跨天去重**：过去 7 天出现过的 → 不再收录

## 当前状态

系统已稳定运行，每日 08:00 自动生成、08:10 审查后推送到飞书。

**持续改进方向**：
- 时间准确性（相对时间计算偶尔不精确）
- 来源多样性（有时过度依赖 TechCrunch）
- 搜索覆盖面（部分信息源可能遗漏）
