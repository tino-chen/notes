## 案例背景

随着自定义技能数量的增加，缺乏统一的发布和管理规范导致维护困难。2026年4月5日，系统性地梳理并制定了完整的技能管理全流程规范，涵盖本地技能管理、GitHub仓库维护、ClawHub发布等多个环节。

## 问题背景

### 原有问题
1. **元数据不统一**：不同技能的SKILL.md元数据字段不一致
2. **命名不规范**：目录名、展示名称、slug之间存在不一致
3. **发布流程混乱**：GitHub和ClawHub发布不同步
4. **依赖管理不清**：外部工具依赖标注不明确
5. **安全审核缺失**：缺乏统一的安全检查标准

### 需求分析
- **标准化**：建立统一的技能发布标准
- **可追溯**：确保每个技能都有完整的来源信息
- **易维护**：建立清晰的维护和更新流程
- **安全性**：建立安全审核和风险控制机制

## 规范体系设计

### 1. 本地技能规范

#### SKILL.md元数据强制字段

| 字段 | 说明 | 示例 | 必填性 |
|------|------|------|--------|
| `name` | 技能slug（kebab-case） | `debunk` | 必填 |
| `description` | 中文一句话描述 | `事实核查与辟谣工具` | 必填 |
| `repository` | GitHub源码链接 | `https://github.com/tino-chen/openclaw-skills/tree/main/debunk` | 必填 |
| `clawhub` | ClawHub发布页链接 | `https://clawhub.com/skills/debunk` | 必填 |
| `guide`（可选） | 笔记站对应文章 | `https://tino-chen.github.io/notes/workflows/debunk.html` | 可选 |

**关键要求**：
- `repository` + `clawhub`是强制要求，每次发布前必须检查
- `guide`有对应笔记时必须添加
- 所有字段必须与实际目录名和发布信息一致

#### 内容规范要求

**内部脚本路径**：
- 必须与实际目录名一致
- 不能残留旧名称或路径错误
- 脚本调用路径需要绝对化

**外部工具依赖标注**：
- OpenClaw内置工具直接标注"内置"
- 第三方MCP工具标注提供方（如"智谱MCP（需配置）"）
- 未配置时必须说明降级行为

**格式规范**：
- 模板中的示例链接用反引号包裹（`` `[原文](URL)` ``）
- 避免被VitePress当成真实链接导致dead link
- 代码示例必须有清晰注释和说明

### 2. GitHub仓库规范

#### README.md规范

**技能顺序固定**：
```
1. note-taking — 笔记创作规范
2. workflow-note — 流程构建类笔记写作
3. daily-ai-briefing — 每日AI进展简报
4. debunk — 事实核查与辟谣工具
5. 新技能往后追加
```

**条目格式规范**：
```markdown
- 📝 Note Taking — 一句话定位
  - 功能点1：详细描述
  - 功能点2：详细描述
  - 功能点3：详细描述
  - 功能点4：详细描述
  - 功能点5：详细描述
```

**ClawHub链接格式**：
- 使用**展示名称 + 可点击链接**
- 不使用slug：`- ClawHub：[Note Taking](https://clawhub.com/skills/note-taking)`
- 保持格式统一和美观

**安装命令同步**：
- 每次更新都必须同步更新安装命令
- 确保命令的正确性和时效性

#### 目录命名规范

**一致性要求**：
- 本地目录 `~/.openclaw/skills/<slug>/` 
- GitHub目录名 `<slug>/`
- 三者必须完全一致

**重命名流程**：
1. 确定新的slug和展示名称
2. 更新本地目录名
3. 更新GitHub目录名
4. 更新SKILL.md中所有相关路径
5. 重新发布到ClawHub

### 3. ClawHub发布规范

#### 展示名称对照表

| slug | 展示名称 |
|------|----------|
| `note-taking` | Note Taking |
| `workflow-note` | Workflow Note |
| `daily-ai-briefing` | Daily AI Briefing |
| `debunk` | Debunk |

#### 发布流程规范

**版本管理**：
- 每次内容更新后必须同步发布新版本
- 版本号递增（x.y.z格式）
- changelog必须描述具体改动内容

**发布命令**：
```bash
clawhub publish ~/.openclaw/skills/<slug> \
  --slug <slug> \
  --name "<Title Case>" \
  --version <x.y.z> \
  --changelog "具体改动描述"
```

### 4. 安全规范

#### 脚本类技能安全要求

**输入验证**：
- 接受用户输入URL的脚本必须拦截：
  - localhost / 127.0.0.1
  - 内网IP（192.168.x.x, 10.x.x.x等）
  - .local域名
  - 非http协议

**依赖管理**：
- Playwright等重量级依赖必须在文档中说明安装步骤
- 提供详细的安装和配置指南
- 说明可能的兼容性问题

#### 审核流程

**安全检查清单**：
- [ ] 输入验证机制完备
- [ ] 网络访问限制合理
- [ ] 依赖项安全性评估
- [ ] 用户数据保护措施

**审核意见回应**：
- 对审核意见逐一回应
- 落实安全改进措施
- 保持审核记录的透明性

## 三端同步检查清单

每次技能更新后，按顺序执行以下检查：

```markdown
## 技能更新同步检查清单

### 1. SKILL.md元数据检查
- [ ] 元数据完整（repository + clawhub + guide）
- [ ] 内部路径与目录名一致
- [ ] 外部依赖标注来源和降级行为

### 2. 内容规范性检查  
- [ ] 格式模板规范（链接包裹）
- [ ] 示例数据已替换为真实数据
- [ ] 脚本路径正确且完整

### 3. GitHub仓库同步
- [ ] README.md描述更新
- [ ] ClawHub链接更新
- [ ] 安装命令更新
- [ ] git commit & push

### 4. ClawHub发布
- [ ] 版本号递增
- [ ] changelog描述准确
- [ ] 发布命令执行成功
```
