## 概述

对OpenClaw记忆系统进行全面重构，区分情形记忆与反思记忆，建立定时任务审查晋升机制，从理论知识导向转为AI协作实践导向。

## 背景

### 原系统问题

**记忆管理混乱：**
- 情形记忆（每日记录）和反思记忆（经验教训）混在一起
- 缺乏系统性整理和经验提炼机制
- 核心配置文件语言不统一（中英文混合）

**用户反馈问题：**
- "口头承诺但没有实际行动"（口头答应但不执行）
- 回复语气太生硬，不像活人
- 快思考导致空承诺

**内容导向问题：**
- 笔记站偏重理论知识，缺乏实用协作经验
- 缺乏AI协作的实操指南和最佳实践

## 解决方案

### 记忆系统重构

**双重记忆架构：**

**📝 情形记忆（Episodic Memory）**
- **位置**：`memory/YYYY-MM-DD.md`（今日 + 昨天）
- **内容**：日常事件、协作记录、任务执行情况
- **目的**：保证连续性，记录每天发生的事情
- **规则**：只记事实（发生了什么），不记反思（学到了什么）

**🧠 反思记忆（Reflective Memory）**
- **位置**：`.learnings/`
- **内容**：`LEARNINGS.md`（被纠正的错误、更好的做法）
- **内容**：`ERRORS.md`（命令失败、API报错、异常行为）
- **内容**：`FEATURE_REQUESTS.md`（用户期望但尚不存在的功能）
- **目的**：积累经验，通过晋升机制让自己越来越聪明
- **晋升**：每天夜里3:00定时任务审查，符合条件的晋升到核心配置文件

### 定时任务体系

**任务调度：**
```yaml
任务时间: 功能
02:00: 笔记草稿撰写（协作回顾与沉淀）
02:30: 笔记审核发布（草稿质量检查）
03:00: 反思记忆审查晋升（经验提炼）
```

**晋升机制：**
```yaml
经验类型 → 晋升目标
行为模式改进 → SOUL.md
工作流程优化 → AGENTS.md  
工具使用坑点 → TOOLS.md
用户偏好 → USER.md
```

### 核心配置文件统一

**语言规范：**
- 所有核心配置文件统一使用中文
- IDENTITY.md内容合并到SOUL.md
- 保持文件结构清晰，便于维护

## 内容导向转型

### 从理论到实践

**原导向：** 理论知识导向
- AI技术原理
- 工具使用说明  
- 理论框架

**新导向：** AI协作实践导向
- 协作流程设计
- 实际问题解决
- 经验沉淀总结

**笔记分类体系：**
```
notes/
├── practice/（工具实战）
│   ├── 具体工具使用案例
│   └── 工具选择对比分析
├── building/（系统构建）  
│   ├── 系统架构设计
│   └── 集成方案实践
└── insights/（经验沉淀）
    ├── 协作模式总结
    └── 最佳实践提炼
```

### 用户反馈响应

**推理模式开启：**
- **问题**：快思考导致口头承诺不行动
- **解决**：开启推理模式（/reasoning）
- **SOUL.md更新**：新增"知道怎么做就立刻做，别走流程"

**回复风格调整：**
- **问题**：回复语气太生硬，不像活人
- **解决**：更新SOUL.md风格章节
- **新增原则**：像活人一样聊天，不要像搜索引擎

## 实施效果

### 记忆管理效率提升

**数据组织：**
- 情形记忆与反思记忆清晰分离
- 避免信息重复和混淆
- 便于快速查找历史记录

**经验提炼：**
- 自动审查机制确保经验不被遗忘
- 晋升到核心配置文件形成长期指导
- 定时任务执行无需人工干预

### 协作实践改善

**实际行动力：**
- 推理模式减少空承诺
- "知道怎么做就立刻做"原则落地
- 用户反馈快速响应

**内容质量：**
- 笔记从理论转为实用协作经验
- 三分类体系便于知识管理
- 实际案例丰富，参考价值高

## 技术实现

### 目录结构
```bash
~/.openclaw/workspace/
├── memory/                 # 情形记忆
│   ├── 2026-04-03.md
│   ├── 2026-04-04.md
│   └── ...
├── .learnings/            # 反思记忆
│   ├── LEARNINGS.md
│   ├── ERRORS.md
│   └── FEATURE_REQUESTS.md
└── notes/                 # 笔记系统
    ├── _drafts/          # 草稿目录
    ├── practice/         # 工具实战
    ├── building/         # 系统构建
    └── insights/         # 经验沉淀
```

### 定时任务配置
```yaml
# 每日3:00执行
- 任务：反思记忆审查晋升
- 输入：.learnings/下的pending条目
- 处理：评估是否符合晋升条件
- 输出：晋升到对应配置文件，标记原条目为promoted
```

### 晋升评估标准
```yaml
评估维度：价值、通用性、稳定性
晋升条件：
1. 解决普遍性问题（非个人特例）
2. 具有长期参考价值
3. 方法论验证有效
4. 不与其他规则冲突
```

## 经验总结

### 设计原则

1. **功能分离**：不同类型记忆使用不同存储机制
2. **自动化**：定时任务减少人工干预
3. **实用性导向**：内容从理论转向实践
4. **用户反馈驱动**：及时响应和调整

### 实施要点

1. **语言统一**：核心配置文件使用中文
2. **分类清晰**：笔记按主题和用途分类
3. **及时记录**：重要事件立即写入对应文件
4. **定期审查**：反思记忆确保经验不流失

### 避免的误区

1. **记忆过度**：只记录重要事实，不记录琐事
2. **分析缺失**：情形记忆只记录不分析
3. **规则僵化**：经验保留更新空间
4. **忽视用户反馈**：持续优化交互体验
